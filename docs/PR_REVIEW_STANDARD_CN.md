# PR 审核标准

本文档定义 `agentmatrix-research` 的 Pull Request 审核标准。

适用对象包括：

- 维护者
- 研究员
- 实习生
- Hermes 等 AI agent

审核的目标不是只判断代码“能不能跑”，而是判断该 PR 是否应该进入统一因子研究后端主线。

## 审核北极星

每条 PR 都要围绕仓库主目标来判断：

- 构建统一的因子研究后端
- 支持可复现的因子实现
- 支持真值对齐验证与 proof 产物
- 支持 Alpha101、Alpha191、Alpha158、Barra 及后续论文因子家族
- 支持 agent 可执行流程，而不是一次性手工脚本

一条 PR 应该让仓库更接近这个目标。

如果一条 PR 只是本地能跑，但不符合统一后端方向，就不应合并。

## 核心审核问题

审核时按顺序回答以下问题：

1. 这条 PR 是否符合仓库使命？
2. 它是在强化统一主线，还是在新建平行体系？
3. 它是否保留了已有可用能力？
4. 它的实现是否技术上正确？
5. 它的验证和证据是否充分？
6. 它的范围是否清晰、可维护、可审查？

前四个问题中，只要有一个答案是否定的，这条 PR 就不应合并。

## 主线规则

本仓库的主线是统一研究后端，不是多个独立原型的集合。

审核时优先支持强化以下部分的 PR：

- `research_core/factor_lab/`
- 共享 operators 与 contracts
- truth、validation、proof、reporting 链路
- 主线使用的 API、service、CLI、runtime

审核时应拒绝或延后以下类型 PR：

- 本应进入 `factor_lab` 却另起一套并行框架
- 重复已有主线逻辑
- 绕开 truth、validation、proof 规范
- 只支持本地 demo、没有接入标准流程

## 合并决策分级

使用以下四种决策。

### 可合并

满足以下条件时，可 `Approve` 或直接合并：

- 没有已知 blocker 或回归
- 与仓库使命一致
- 技术实现正确
- 变更范围清晰
- 针对该类型改动的验证或测试足够

### 留评但不合并

适用于：

- 方向有价值
- 实现还不完整
- 由于 PR 是自己账号提交或探索性质，不适合做正式驳回

这类情况常见于维护者本人账号或 AI agent 代维护者账号提交的 PR。

### 要求修改

出现以下任一具体 blocker 时，应 `Request changes`：

- 运行时错误
- 导入错误
- 语法错误
- API 兼容性破坏
- 因子逻辑错误
- 验证结果失真或误导
- 安全或数据卫生问题
- 流程会把失败伪装成成功

### 作为被覆盖版本关闭

适用于：

- 该 PR 的方向已被后续更完整 PR 覆盖
- 它属于 stacked PR 中的中间版本，不适合单独合并
- 它已经失去独立合并价值

此时应在关闭评论中明确说明由哪条更新 PR 取代。

## 硬阻塞问题

出现以下任一问题时，应立即阻止合并：

- `SyntaxError`、`ImportError` 或导入阶段崩溃
- 破坏现有调用方的函数签名变更
- 未经清晰说明即改变因子语义
- CI 或验证流程会显示绿色但实际失败
- 引用了不存在的符号、文件或 contract
- 泄露 token、secret、账号路径或私有产物
- 将供应商凭证写入仓库文件
- 破坏可复现性或 proof 可追溯性

## 因子类 PR 审核规则

如果 PR 修改了因子逻辑、operator 语义或 truth compare 行为，需要检查：

- 公式意图与来源对齐
- operator 语义是否正确
- 数据字段假设是否成立
- 窗口参数处理是否合理
- 缺失值处理是否合理
- rank 与 neutralize 语义是否一致
- 是否影响已有因子结果

对于因子家族复现：

- 逻辑确实可复用时，优先下沉到共享 operator
- 泛化还不稳定时，允许先保留在局部 helper

不要为了“看起来更抽象”而过早共享化，进而引入回归风险。

## 验证要求

验证深度应和 PR 类型匹配。

### Operator 或因子逻辑改动

期望证据：

- 有针对性的测试或可复现命令
- 样例输出或验证说明
- 没有明显破坏现有调用点

### Truth、proof、reporting、runtime 改动

期望证据：

- proof 或 truth 产物路径
- 生成命令
- 验证范围总结
- 已知缺口说明

### CI 或 workflow 改动

期望证据：

- 全流程路径被审过
- 失败行为明确
- 关键步骤没有无兜底的 `continue-on-error`
- 多 commit PR 场景下的范围检测正确

## 对作者的补充提问

如果上下文不足，应要求作者补充：

- 这条 PR 在解决什么问题
- 为什么应该进主线
- 实际跑了哪些命令
- 产出了哪些测试或 proof
- 可能影响哪些现有行为

如果这些问题都无法明确回答，通常说明 PR 还没准备好。

## Stacked PR 审核规则

如果多个 PR 是叠加关系：

- 先识别最新且最完整的版本
- 除非前一个版本有明确独立价值，否则不要先合中间版本
- 新版本合并后，再关闭被覆盖的旧 PR
- 如果可能，优先合并原 PR 分支，尽量保留作者贡献记录

审核 stacked PR 时，不要完全脱离上下文单独判断每一条。

## 自己账号提交 PR 的规则

如果 PR 是维护者自己账号提交的，GitHub 可能不允许正式 `Request changes`。

此时应：

- 留下明确 `Comment`
- 写清楚当前不应合并
- 根据情况保留 PR 等修复，或直接关闭

没有 `Request changes` 按钮，不代表这条 PR 就能合并。

## 推荐审核顺序

当同时有多条 PR 待处理时，按以下顺序审核：

1. 可以直接合并且主线一致的 PR
2. 已被覆盖、应关闭的 PR
3. 有明确 blocker、需要修改的 PR
4. 方向需重定向的探索型或平行体系 PR

这样能在不牺牲主线稳定性的前提下持续推进仓库演进。

## 标准审核输出

一条好的审核意见至少应包含：

- 合并决策
- 主要原因
- 具体 blocker（如果有）
- 下一步建议动作

示例：

- “可以合并。这条 PR 强化了统一 factor_lab 主线，未发现新增 blocker。”
- “当前不要合并。该改动破坏了现有 `top_k` 调用链，产生运行时兼容性回归。”
- “作为被覆盖版本关闭。相同方向已由更新且更完整的 PR #N 覆盖。”

## 审核结论模板

可使用如下模板：

```text
结论：
- 合并 / 仅评论不合并 / 要求修改 / 作为被覆盖版本关闭

原因：
- 主线一致性
- 正确性
- 回归风险
- 验证充分性

下一步：
- 现在合并
- 修复 blocker 后再提
- 关闭当前 PR，保留更新版本
```

## 最终原则

本仓库不应因为一条 PR：

- 内容很多
- 看起来复杂
- 局部能跑

就直接合并。

真正应合并的 PR 必须满足：

- 正确
- 可复现
- 可审计
- 与统一后端主线一致
- 对未来因子研究与 agent 执行有长期价值
